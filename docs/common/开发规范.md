# Java Windows 桌面应用开发规范

本文件是仓库的最高开发基线，AGENTS.md 基于此文件简化为可直接执行的规则。所有新增、修改、重构与评审工作均以本文档约束为准。

---

## 1. 概述

### 1.1 适用范围

本规范适用于基于 Java 开发的 Windows 10/11 桌面应用程序，涵盖项目结构、分层架构、命名、编码、UI 开发、数据库、多线程、日志、异常处理、打包发布等全流程。

### 1.2 核心原则

- **分层解耦**：UI / Service / DAO 严格分层，单向依赖
- **UI 线程优先**：所有 GUI 操作必须在 UI 线程执行，耗时操作必须异步
- **嵌入式部署**：优先使用嵌入式数据库（SQLite / H2），无需用户额外安装
- **本地路径规范**：数据存储在用户目录（`%APPDATA%` / `%LOCALAPPDATA%`）下，禁止硬编码路径
- **文档驱动**：先设计后编码，文档、代码、数据库脚本必须保持一致

### 3.2 分层职责

```
┌─────────────────────────────────────────┐
│  UI 层 (ui/)                            │
│  - Stage/Scene 管理与窗口跳转            │
│  - 控件事件绑定与用户交互                │
│  - 输入校验（前端级，如非空/格式）       │
│  - 线程调度（将耗时任务提交后台线程）    │
│  - 不得包含业务逻辑与数据库操作          │
├─────────────────────────────────────────┤
│  Service 层 (service/)                  │
│  - 核心业务逻辑实现                      │
│  - 事务管理（通过 DAO 编排）             │
│  - 业务规则校验                          │
│  - 跨模块流程编排                        │
│  - 事件发布与订阅                        │
├─────────────────────────────────────────┤
│  DAO 层 (dao/)                          │
│  - 数据库 CRUD 操作                      │
│  - SQL 语句封装                          │
│  - 连接池 / 连接管理                     │
│  - 不得包含业务判断                      │
│  - 返回 Model 对象，不接受 UI 层对象     │
├─────────────────────────────────────────┤
│  Model 层 (model/)                      │
│  - DO: 与数据库表一一对应                │
│  - VO: 视图展示用对象                    │
│  - DTO: 层间数据传输对象                 │
│  - 纯数据结构，不含业务逻辑              │
└─────────────────────────────────────────┘
```

### 3.3 依赖方向

```
ui/ ──→ service/ ──→ dao/ ──→ model/
                 └──→ model/
          └──→ config/
          └──→ enums/
          └──→ util/
```

不允许反向依赖（如 dao/ 引用 service/，service/ 引用 ui/）。

## 4. 命名规范

### 4.1 通用规则

- 所有源码文件统一使用 `UTF-8` 编码
- 命名采用英文语义，禁止使用拼音或无意义缩写
- 除常量外，不使用下划线分隔 Java 标识符（数据库命名除外）

### 4.2 包命名

```
com.personal.windows.{应用名}.{分层}
```

- 应用名：全小写英文，如 `todo`、`note`、`reminder`
- 分层：`ui` / `service` / `dao` / `model` / `config` / `enums` / `util`

### 4.3 类命名

| 分层 | 模式 | 示例 |
|------|------|------|
| 应用入口 | `{App}Application` 或 `MainApp` | `TodoApplication` |
| UI 控制器 | `{页面名}Controller` | `TodoListController` |
| 自定义控件 | `{控件名}View` / `{控件名}Component` | `TaskCardView` |
| Service 接口 | `I{业务名}Service` | `ITodoService` |
| Service 实现 | `{业务名}ServiceImpl` | `TodoServiceImpl` |
| DAO 接口 | `I{业务名}Dao` | `ITodoDao` |
| DAO 实现 | `{业务名}DaoImpl` | `TodoDaoImpl` |
| 数据库实体 | `{业务名}Do` 或 `{业务名}Entity` | `TodoDo` |
| 视图对象 | `{业务名}Vo` | `TodoVo` |
| 数据传输对象 | `{业务名}Dto` | `TodoQueryDto` |
| 配置类 | `{功能}Config` | `DatabaseConfig` |
| 枚举类 | 业务语义命名 | `PriorityEnum`、`TaskStatus` |
| 工具类 | `{功能}Util` 或 `{功能}Utils` | `DateUtil`、`StringUtils` |
| 异常类 | `{业务}Exception` | `DataAccessException` |

### 4.4 方法命名

| 操作类型 | 前缀 | 示例 |
|---------|------|------|
| 查询单条 | `get` / `find` | `getUserById`, `findByName` |
| 查询列表 | `list` / `query` | `listTodos`, `queryByFilter` |
| 新增 | `add` / `create` / `save` | `addTodo`, `createUser` |
| 修改 | `update` / `modify` | `updateTodo` |
| 删除 | `delete` / `remove` | `deleteTodo`, `removeById` |
| 批量操作 | `batch{操作}` | `batchDelete`, `batchUpdate` |
| 布尔判断 | `is` / `has` / `can` | `isCompleted`, `hasPermission` |

### 4.5 资源文件命名

| 类型 | 命名规则 | 示例 |
|------|---------|------|
| FXML 文件 | 与 Controller 同名 | `TodoList.fxml` |
| CSS 文件 | 小写连字符 | `main-style.css`、`dark-theme.css` |
| 图片/图标 | `{用途}-{尺寸}.{扩展名}` | `app-icon-64.png`、`add-btn-24.png` |
| 配置文件 | `application.properties` | — |
| 国际化资源 | `messages_{locale}.properties` | `messages_zh_CN.properties` |

### 4.6 数据库命名

| 对象 | 规则 | 示例 |
|------|------|------|
| 表名 | 全小写，下划线分隔，单数名词 | `todo_item`、`note_category` |
| 字段名 | 全小写，下划线分隔 | `created_time`、`assignee_id` |
| 主键 | 统一 `id`，自增整数 | `id INTEGER PRIMARY KEY AUTOINCREMENT` |
| 外键 | `{关联表名}_id` | `category_id`、`user_id` |
| 索引 | `idx_{表名}_{字段}` | `idx_todo_item_status` |
| 公共字段 | 建议每表包含 | `created_time`、`updated_time` |

## 5. UI 开发规范

### 5.1 JavaFX 应用架构

```
Application.launch()
    └─ start(Stage primaryStage)
        └─ Scene / FXML 加载
            └─ Controller.initialize()
                └─ 初始化控件、绑定事件
```

### 5.2 FXML 编写规范

- FXML 文件与对应 Controller 类放在同包路径下（如 `ui/` 包），资源文件放在 `resources/fxml/` 下
- 使用 `fx:controller` 指定对应的 Controller 全限定类名
- 控件 `fx:id` 使用小驼峰命名，与 Controller 中 `@FXML` 注解的字段名一致
- 事件处理方法命名：`on{控件名}{事件}`，如 `onSaveButtonClick`
- 禁止在 FXML 中内嵌脚本代码

```xml
<!-- 示例 -->
<VBox xmlns="http://javafx.com/javafx"
      xmlns:fx="http://javafx.com/fxml"
      fx:controller="com.personal.windows.todo.ui.TodoListController">
    <children>
        <ListView fx:id="todoListView" />
        <Button text="添加" onAction="#onAddButtonClick" />
    </children>
</VBox>
```

### 5.3 Controller 编写规范

- Controller 类仅负责：
    - 控件初始化（`initialize()` 方法）
    - 响应用户交互事件（点击、选择、输入）
    - 调用 Service 层方法
    - 将 Service 返回数据绑定到 UI 控件
    - 窗口跳转与关闭
- Controller 中禁止：
    - 直接创建数据库连接或执行 SQL
    - 包含业务规则判断
    - 直接操作文件 IO
- 使用 `@FXML` 注解绑定控件和事件方法
- Controller 中注入 Service 实例，通过构造函数或 setter 注入

### 5.4 窗口管理

- 主窗口由 `{App}Application` 管理
- 子窗口/弹窗通过 Stage 管理，推荐使用 Modal 模式（`stage.initModality(Modality.APPLICATION_MODAL)`）
- 窗口生命周期必须明确：打开 → 使用 → 关闭并释放资源
- 避免同时打开过多窗口（建议不超过 3 层弹窗嵌套）

### 5.5 CSS 规范

- 样式类命名使用小写连字符（kebab-case），如 `.todo-item`、`.header-title`
- 样式类命名体现语义而非外观，如用 `.priority-high` 而非 `.red-text`
- 主题切换通过切换 CSS 文件实现，不在代码中硬编码颜色
- 优先使用 `styleClass` 而非内联 `setStyle()`

### 5.6 响应式与国际化

- 窗口最小尺寸必须明确定义（`setMinWidth` / `setMinHeight`）
- 布局自适应窗口缩放，禁止使用绝对定位（除特殊场景外）
- 所有用户可见文本使用国际化资源文件（`ResourceBundle`），禁止硬编码中文/英文

## 6. Service 层开发规范

### 6.1 基本原则

- 每个 Service 接口对应一个业务领域
- Service 实现类使用 `@Service` 或自定义注解标记（或通过工厂类管理）
- Service 方法是业务操作的最小事务单元
- 复杂流程拆分为多个私有子方法，每个子方法职责单一

### 6.2 方法设计

```java
// 接口示例
public interface ITodoService {

    List<TodoVo> listTodos(TodoQueryDto query);

    TodoVo getTodoById(long id);

    TodoVo addTodo(TodoDto dto);

    TodoVo updateTodo(long id, TodoDto dto);

    void deleteTodo(long id);

    void batchDelete(List<Long> ids);
}
```

- 查询方法返回 VO 或 DTO，不直接暴露 DO
- 写操作接收 DTO，返回更新后的 VO
- 批量操作必须带事务（手动管理或通过 DAO 层 ensureTransaction）

### 6.3 事务管理

- 数据库操作统一通过 DAO 层执行
- 涉及多表写入的操作，Service 负责开启/提交/回滚事务
- 嵌入式数据库（SQLite）默认自动提交，多步写入需显式管理事务

```java
// 典型事务模式（SQLite）
public void batchUpdateStatus(List<Long> ids, Status newStatus) {
    Connection conn = dataSource.getConnection();
    try {
        conn.setAutoCommit(false);
        for (Long id : ids) {
            todoDao.updateStatus(conn, id, newStatus);
        }
        conn.commit();
    } catch (Exception e) {
        conn.rollback();
        throw new ServiceException("批量更新失败", e);
    } finally {
        conn.setAutoCommit(true);
    }
}
```

### 6.4 异常处理

- Service 层统一抛出业务异常（`ServiceException` / `BusinessException`）
- 不允许将 SQLException 直接抛出到 UI 层
- 异常信息包含可读的中文描述，用于 UI 提示
- 所有异常必须记录日志

## 7. DAO 层与数据库规范

### 7.1 连接管理

- 使用连接池管理数据库连接（HikariCP 或 SQLite 单连接模式）
- 数据库文件统一存放在用户数据目录：`System.getProperty("user.home")` 或 `%APPDATA%`
- 启动时自动检测数据库文件是否存在，不存在则自动创建并执行初始化脚本

### 7.2 SQL 编写规范

- SQL 关键字大写，表名/字段名小写
- 禁止拼接 SQL 字符串，必须使用 `PreparedStatement` 参数化查询
- 复杂查询优先使用 JOIN，避免 N+1 查询
- 每张表必须有对应的建表 SQL 脚本，存放在 `src/main/resources/db/`

```sql
-- 示例：建表脚本 V1__create_todo_item.sql
CREATE TABLE IF NOT EXISTS todo_item (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    title       TEXT    NOT NULL,
    description TEXT,
    priority    TEXT    NOT NULL DEFAULT 'MEDIUM',
    status      TEXT    NOT NULL DEFAULT 'PENDING',
    due_date    TEXT,
    created_time TEXT   NOT NULL DEFAULT (datetime('now', 'localtime')),
    updated_time TEXT   NOT NULL DEFAULT (datetime('now', 'localtime'))
);
```

### 7.3 数据库版本管理

- 所有 DDL 脚本按版本号命名：`V{版本号}__{描述}.sql`（如 `V1__init.sql`）
- 使用 Flyway 或自定义迁移工具管理版本
- 禁止手动修改已发布的迁移脚本，变更必须新建迁移脚本

### 7.4 DAO 实现规范

```java
// 典型 DAO 实现
public class TodoDaoImpl implements ITodoDao {

    private final DataSource dataSource;

    public TodoDaoImpl(DataSource dataSource) {
        this.dataSource = dataSource;
    }

    @Override
    public List<TodoDo> listByStatus(Status status) {
        String sql = "SELECT * FROM todo_item WHERE status = ? ORDER BY created_time DESC";
        try (Connection conn = dataSource.getConnection();
             PreparedStatement ps = conn.prepareStatement(sql)) {
            ps.setString(1, status.name());
            try (ResultSet rs = ps.executeQuery()) {
                List<TodoDo> list = new ArrayList<>();
                while (rs.next()) {
                    list.add(mapRow(rs));
                }
                return list;
            }
        } catch (SQLException e) {
            throw new DataAccessException("查询失败", e);
        }
    }

    private TodoDo mapRow(ResultSet rs) throws SQLException {
        TodoDo item = new TodoDo();
        item.setId(rs.getLong("id"));
        item.setTitle(rs.getString("title"));
        // ... 其他字段
        return item;
    }
}
```

- 每个 DAO 方法中 ResultSet / PreparedStatement / Connection 必须使用 try-with-resources 关闭
- 统一抽取 `mapRow(ResultSet)` 方法进行结果映射
- 数据库字段名下划线到 Java 属性小驼峰的转换在 `mapRow` 中手动处理或使用框架自动映射

## 8. 异常处理与日志规范

### 8.1 异常分类

| 异常类型 | 用途 | 示例 |
|---------|------|------|
| `BusinessException` | 业务规则违反 | 重复创建、状态不允许、权限不足 |
| `DataAccessException` | 数据库操作失败 | 连接失败、SQL 执行错误 |
| `ValidationException` | 输入校验失败 | 必填字段为空、格式不正确 |
| `ConfigException` | 配置错误 | 配置文件缺失、格式错误 |

### 8.2 异常处理原则

- 各层只处理本层能处理的异常，不能处理的向上抛出
- UI Controller 层统一 catch 业务异常，通过弹窗/提示栏告知用户
- 全局未捕获异常通过 `Thread.setDefaultUncaughtExceptionHandler` 兜底
- 异常消息对用户友好（中文），技术细节写入日志

### 8.3 日志规范

- 统一使用 SLF4J 门面 + Logback 实现
- 日志级别使用规则：

| 级别 | 使用场景 |
|------|---------|
| ERROR | 系统错误、必须人工介入的问题 |
| WARN | 可恢复的异常、降级处理、配置不当 |
| INFO | 关键业务流程节点、启动/关闭、配置加载 |
| DEBUG | 调试信息、方法入参出参、SQL 参数 |

- 日志输出位置：开发环境输出到控制台，生产环境输出到用户数据目录下的 `logs/` 文件夹
- 禁止在日志中输出密码、Token 等敏感信息
- 异常日志必须包含完整堆栈：`log.error("操作失败", exception)`

## 9. 多线程与并发规范

### 9.1 UI 线程规则

- **JavaFX**：所有 UI 操作必须在 JavaFX Application Thread 中执行，使用 `Platform.runLater()` 从后台线程更新 UI
- **Swing**：所有 UI 操作必须在 Event Dispatch Thread (EDT) 中执行，使用 `SwingUtilities.invokeLater()`
- 禁止在 UI 线程中执行：
    - 网络请求（HTTP、Socket 等）
    - 数据库操作
    - 文件 IO（大文件）
    - 复杂计算

### 9.2 后台任务执行

- 简单任务：`CompletableFuture.supplyAsync()` + `Platform.runLater()`
- 需要进度反馈：JavaFX `Task` + `ProgressIndicator`
- 定时任务：`ScheduledExecutorService` 或 `java.util.Timer`
- 推荐使用 `Executors.newFixedThreadPool(Runtime.getRuntime().availableProcessors())` 创建线程池

```java
// JavaFX 典型异步模式
CompletableFuture.supplyAsync(() -> todoService.listTodos(query))
    .thenAcceptAsync(todos -> {
        todoListView.getItems().setAll(todos);
    }, Platform::runLater)
    .exceptionally(ex -> {
        Platform.runLater(() -> showError("加载失败: " + ex.getMessage()));
        return null;
    });
```

### 9.3 线程安全

- Service 层应为无状态或线程安全，便于多线程调用
- DAO 层的 DataSource 实例线程安全，Connection 非线程安全——每个线程/操作独立获取和关闭
- 共享状态使用 `AtomicReference`、`ConcurrentHashMap` 等并发集合
- 窗口关闭时必须取消关联的后台任务，避免内存泄漏

## 10. 配置管理规范

### 10.1 配置文件

- 优先使用 `application.properties` 作为主配置文件
- 配置项命名使用小写字母 + 点号分隔：`app.window.title`、`db.url`
- 按用途分组：

```properties
# 应用配置
app.name=PersonalTodo
app.version=1.0.0
app.window.width=800
app.window.height=600

# 数据库配置
db.type=sqlite
db.path=${user.home}/.personalapp/data.db

# 日志配置
log.level=INFO
log.path=${user.home}/.personalapp/logs
```

### 10.2 数据存储路径

- 应用数据统一存储在 `%APPDATA%\{应用名}\` 或 `%LOCALAPPDATA%\{应用名}\`
- 通过 `System.getenv("APPDATA")` 或 `System.getProperty("user.home")` 动态获取
- 首次启动时自动创建所需目录结构
- 禁止在安装目录下写入运行时数据（Program Files 通常无写权限）

## 11. 资源管理规范

### 11.1 资源加载

- FXML、CSS、图片等资源通过 `getClass().getResource()` / `ClassLoader.getResource()` 加载
- 资源路径使用绝对路径（以 `/` 开头从 classpath 根开始）
- 大型资源文件延迟加载，避免阻塞启动

### 11.2 国际化 (i18n)

- 所有面向用户的文本从 `ResourceBundle` 获取
- 资源文件命名：`messages_{language}_{country}.properties`
- 默认语言为简体中文（`messages_zh_CN.properties`）
- 代码中禁止硬编码用户可见字符串

```java
ResourceBundle bundle = ResourceBundle.getBundle("i18n/messages");
String title = bundle.getString("app.title");
```

## 12. 测试规范

### 12.1 测试分层

| 层级 | 框架 | 覆盖要求 |
|------|------|---------|
| 单元测试 (Service/Util) | JUnit 5 + Mockito | 核心业务逻辑覆盖率 ≥ 70% |
| DAO 集成测试 | JUnit 5 + 内存数据库 (H2) | 关键 SQL 场景全覆盖 |
| UI 测试 | TestFX（JavaFX）| 关键用户路径覆盖 |

### 12.2 测试原则

- Service 层测试使用 Mock DAO，隔离数据库依赖
- DAO 层集成测试使用 H2 内存数据库，避免依赖真实 SQLite 文件
- 测试数据和测试后清理在 `@BeforeEach` / `@AfterEach` 中完成
- 测试类命名：`{被测类名}Test`
- 测试方法命名：`{被测方法}_{场景}_{预期结果}`，如 `addTodo_duplicateTitle_throwsException`

## 13. 打包与发布规范

### 13.1 打包流程

1. Maven 编译打包：`mvn clean package`
2. 使用 Inno Setup 创建 Windows 安装包（.exe）：

- MINOR：向下兼容的功能新增

- [ ] 所有单元测试通过

- [ ] 版本号与 changelog 一致

## 14. Git 工作流规范

### 14.1 分支策略

```
main          ← 发布分支，仅通过 PR 合并
  └─ dev      ← 开发分支
       ├─ feature/{应用名}-{功能描述}   ← 功能开发
       ├─ bugfix/{描述}                 ← Bug 修复

```
<类型>(<范围>): <简短描述>

类型：`feat` / `fix` / `docs` / `style` / `refactor` / `test` / `chore`

feat(todo): 添加待办事项拖拽排序功能

支持在列表中通过拖拽调整待办事项顺序

关联: #12

- 代码已本地编译通过
- 相关单元测试通过
- 没有提交 IDE 配置文件（`.idea/`、`*.iml` 等已在 `.gitignore`）
- 没有提交编译产物（`target/`、`build/`）
- 没有提交日志文件、数据库文件

## 15. 代码质量规范

### 15.1 编码风格

- 缩进：4 个空格，禁止使用 Tab
- 单行最大长度：120 字符
- 大括号：K&R 风格（左括号不换行）
- 导入语句：不使用通配符（`import java.util.*` → 逐类导入）
- 空行：逻辑块之间使用空行分隔，方法之间一个空行

### 15.2 注释规范

- 公开 API（public 方法/类）必须写 Javadoc
- 复杂逻辑必须写行内注释说明意图，而非复述代码
- 不写无意义注释（如 `// 设置名称` 对应 `setName()`）
- 待完成事项使用 `// TODO: 描述`，需关联 issue 编号

### 15.3 静态检查

- 推荐使用 Checkstyle / SpotBugs 进行静态代码分析
- Maven 配置 checkstyle 插件，构建时自动检查
- 严重程度为 Error 的规则必须修复，Warning 级别建议修复

## 附录：常用工具方法参考

### A. 获取应用数据目录

```java
public class AppPathUtil {
    private static final String APP_NAME = "PersonalApp";

    public static Path getDataDir() {
        String appData = System.getenv("APPDATA");
        if (appData == null) {
            appData = System.getProperty("user.home") + "/AppData/Roaming";
        }
        return Path.of(appData, APP_NAME);
    }

    public static Path getDbPath() {
        return getDataDir().resolve("data.db");
    }

    public static Path getLogPath() {
        return getDataDir().resolve("logs");
    }
}
```

### B. 确保目录存在

```java
public static void ensureDir(Path dir) throws IOException {
    if (!Files.exists(dir)) {
        Files.createDirectories(dir);
    }
}
```
